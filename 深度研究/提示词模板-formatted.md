**研究主题：** {{在此填写一个 AI Agent 安全子主题 / 子领域}}

你是一名 AI Agent 安全首席研究员、安全架构师、红蓝对抗专家和企业落地顾问。请围绕本次研究主题，输出一份可用于**真实组织决策、工程落地、风险治理和安全建设**的深度研究报告。

## 一、研究边界

本报告必须完全置于 **AI Agent 安全** 大框架下展开。不要把本次研究主题泛化成普通大模型安全、普通应用安全、普通网络安全或抽象 AI 伦理讨论。

这里的 AI Agent 指具备以下一个或多个能力的系统：接收目标、规划任务、调用工具 / API / 浏览器 / 代码执行环境 / 数据库 / 企业系统、访问或写入记忆、与其他 Agent 协作、代表用户执行动作、影响数字或物理环境。

> **研究重点不是“模型会不会回答错”**，而是：当 Agent 具备行动能力、权限、工具、记忆、上下文、身份和自主性后，会如何被滥用、被劫持、失控、泄露数据、扩大权限、污染环境、触发级联故障，以及如何实战防御。

## 二、资料源要求

优先使用一手、官方、权威、可验证资料。每个关键判断必须有来源、发布日期、适用范围和可信度说明。

**资料优先级如下：**

1. 一级资料：OWASP GenAI Security Project、OWASP Top 10 for Agentic Applications、OWASP Agentic AI Threats and Mitigations、NIST AI RMF / NIST GenAI Profile、MITRE ATLAS、CISA / NSA / FBI / NCSC / ENISA 等官方安全指南。
2. 二级资料：MCP 官方规范与安全最佳实践、CoSAI、ISO / IEC、CIS、云厂商与模型厂商官方安全文档、系统卡、安全公告、红队报告。
3. 三级资料：NVD / CVE、GitHub Security Advisories、厂商漏洞公告、真实事件复盘、可信安全实验室报告。
4. 四级资料：高质量论文、会议论文、arXiv 预印本、行业研究。必须区分“已验证事实”和“研究假设”。
5. 谨慎使用：媒体报道、博客、二手解读、营销白皮书。只能作为线索，不能作为核心论据。

必须检索最新资料，标注“截至日期”。若资料冲突，优先官方原文、漏洞公告、标准文件和可复现实证。不得编造引用。找不到证据时，直接写“暂无可靠证据”。

## 三、研究目标

**请回答以下核心问题：**

1. 本次研究主题到底解决 AI Agent 安全中的哪个真实问题？
2. 它在真实业务、真实系统、真实攻防中为什么重要？
3. 当前主流技术、框架、产品和组织实践做到什么程度？
4. 主要攻击面、失效模式、风险链路和误用场景是什么？
5. 哪些风险是高概率，哪些是低概率高影响，哪些只是概念炒作？
6. 现有防御方案中，哪些可落地，哪些成本高，哪些存在明显缺口？
7. 一个真实企业要落地防护，应如何设计架构、流程、权限、监控、评估和应急？
8. 未来 6-18 个月，本次研究主题可能如何演进？

## 四、分析框架

请至少从以下层次展开，不要机械罗列，要围绕本次研究主题取舍、合并和深入。

1. 概念层：定义、边界、相关概念、易混淆概念、与普通 LLM 安全的差异。
2. 场景层：真实业务场景、Agent 形态、用户角色、系统边界、数据流、动作流、权限流。
3. 架构层：模型、编排器、工具层、MCP / 插件 / 连接器、RAG、记忆、身份系统、权限系统、审计系统、沙箱、网关、策略引擎、监控系统。
4. 威胁层：目标劫持、提示注入、工具滥用、权限滥用、数据泄露、记忆投毒、RAG 投毒、供应链污染、代码执行、跨 Agent 传播、级联故障、信任操纵、失控 Agent。
5. 攻击路径层：攻击者入口、前置条件、利用链、可观测信号、影响范围、横向移动或权限放大路径。只描述防御研究所需的高层路径，不输出可直接用于攻击的操作步骤、代码或绕过细节。
6. 控制层：预防、检测、响应、恢复、治理。每个控制项必须说明拦截什么风险、放在哪里、由谁负责、如何验证有效。
7. 工程层：系统设计、权限模型、隔离策略、日志字段、告警规则、评估集、CI / CD 安全门禁、运行时策略、回滚机制。
8. 组织层：Owner、RACI、审批流、例外流程、供应商管理、红队节奏、合规映射、预算优先级。
9. 商业层：成本、收益、误报率、延迟、用户体验、上线阻力、运维复杂度、建设优先级。
10. 趋势层：标准演进、协议演进、模型能力变化、Agent 平台化、监管变化、攻防自动化。

## 五、必须覆盖的实战问题

请优先选择与本次研究主题最相关的 3-5 个真实场景深入分析，例如：

* 企业知识库 / RAG Agent
* 浏览器 / Computer Use Agent
* 代码生成 / DevOps Agent
* SOC / SecOps Agent
* 客服 / 销售 / 财务 / HR Agent
* MCP 工具生态 / 插件生态
* 多 Agent 协作系统
* 数据分析 Agent
* 工作流自动化 Agent
* OT / 关键基础设施 Agent

**每个场景必须输出：**

* 业务目标
* Agent 能做什么动作
* 涉及哪些敏感数据和关键资产
* 信任边界在哪里
* 最可能发生的攻击或失效模式
* 最坏影响是什么
* 当前常见防线为什么可能不够
* 推荐的最小可行防线
* 推荐的成熟防线
* 检测信号和审计证据
* 上线前必须通过的安全验收项

## 六、研究方法

**按以下流程完成研究：**

1. 明确本次研究主题的定义、范围、排除项和核心假设。
2. 收集并筛选权威资料，形成“证据矩阵”：来源、发布日期、机构类型、核心观点、适用性、局限性。
3. 建立 AI Agent 系统模型：资产、身份、权限、工具、上下文、记忆、数据流、动作流、外部依赖。
4. 建立威胁模型：攻击者、入口、利用链、影响、检测点、防护点。
5. 映射主流框架：OWASP Agentic Top 10、OWASP LLM Top 10、MITRE ATLAS、NIST AI RMF、MCP 安全最佳实践、零信任、安全开发生命周期。
6. 提取真实案例、公开漏洞、红队发现、产品限制和行业落地经验。
7. 评估现有防御：有效性、覆盖面、成熟度、工程成本、误报漏报、部署前提。
8. 给出落地架构：从最小可行防线到成熟防线，分阶段建设。
9. 设计验证方案：红队测试、自动化评估、上线验收、运行时监控、复盘机制。
10. 输出结论：该做什么、不该做什么、先做什么、哪些风险必须接受或转移。

## 七、报告结构

请按以下结构输出：

### 1. 执行摘要

用极短篇幅说明：

* 本次研究主题的核心结论
* 最大风险
* 最优先落地的 3 项措施
* 最不建议做的 3 件事
* 适合立即行动的组织类型
* 不适合盲目投入的场景

### 2. 本次研究主题定义与边界

说明：

* 本次研究主题是什么
* 不是什么
* 为什么属于 AI Agent 安全
* 与普通 LLM 安全、传统 AppSec、云安全、数据安全的关系
* 本报告的假设、限制和适用范围

### 3. 权威资料与行业共识

**输出“证据矩阵”：**

| 资料来源 | 类型 | 日期 | 核心观点 | 与本次研究主题的关系 | 可信度 | 局限 |
| ---- | -- | -: | ---- | ---------- | --- | -- |

必须总结当前行业共识和仍无共识的问题。不得只堆引用。

### 4. 真实场景拆解

输出 3-5 个关键场景：

| 场景 | Agent 能力 | 关键资产 | 信任边界 | 主要风险 | 业务影响 | 当前防线缺口 |
| -- | -------- | ---- | ---- | ---- | ---- | ------ |

随后深入解释每个场景的真实风险链路。

### 5. 系统架构与攻击面

画出文字版参考架构：

用户 / 外部内容 / Agent 编排器 / 模型 / 工具层 / MCP 或插件 / RAG / 记忆 / 权限系统 / 数据源 / 执行环境 / 日志监控 / 策略引擎 / 人类审批。

逐层说明：

* 资产
* 信任假设
* 可被操纵的输入
* 可被滥用的动作
* 防御控制点
* 必要日志

### 6. 威胁模型

输出：

| 威胁 | 攻击入口 | 前置条件 | 典型链路 | 影响 | 可观测信号 | 映射框架 | 防护优先级 |
| -- | ---- | ---- | ---- | -- | ----- | ---- | ----- |

必须覆盖与本次研究主题直接相关的 Agent 特有风险，例如目标劫持、工具滥用、权限滥用、记忆污染、上下文污染、跨 Agent 传播、失控行为、供应链污染、不可逆动作误执行。

### 7. 防御架构

按“最小可行防线 / 标准企业防线 / 高成熟度防线”三档输出。

每项控制必须包含：

| 控制项 | 防什么 | 放在哪里 | 实现方式 | Owner | 验收标准 | 代价 / 副作用 |
| --- | --- | ---- | ---- | ----- | ---- | -------- |

优先考虑以下原则：

* 最小权限
* 最小自主性
* 最小上下文暴露
* 工具白名单
* 动作级授权
* 高风险操作人类确认
* 短效凭证
* 意图绑定
* 会话隔离
* 记忆隔离
* 工具调用审计
* 沙箱执行
* 输出校验
* 外部内容不可信
* 默认拒绝
* 爆炸半径限制
* Kill Switch
* 不可篡改日志
* 持续红队与回归评估

### 8. 红队、评估与验收

设计防御性测试方案，不提供攻击代码。

至少包含：

| 测试项 | 测什么风险 | 测试方法 | 成功 / 失败判定 | 指标 | 频率 |
| --- | ----- | ---- | --------- | -- | -- |

指标必须量化，例如：

* 间接提示注入成功率
* 未授权工具调用率
* 敏感数据泄露率
* 禁止动作执行率
* 人类审批绕过率
* 记忆污染持久化率
* RAG 污染命中率
* 权限越界调用率
* 沙箱逃逸告警率
* MTTD / MTTR
* 审计覆盖率
* 高风险动作阻断率

### 9. 落地路线图

按阶段输出：

| 阶段 | 时间 | 目标 | 任务 | 交付物 | Owner | 依赖 | 风险 |
| -- | -: | -- | -- | --- | ----- | -- | -- |

必须包含：

* 0-30 天：资产盘点、Agent 清单、权限收敛、日志补齐、最高风险场景封堵。
* 31-60 天：工具网关、策略引擎、人类审批、红队评估集、MCP / 插件治理。
* 61-90 天：运行时监控、自动化评估、供应链校验、事件响应预案。
* 90 天以上：成熟度模型、持续红队、合规映射、平台化治理。

### 10. Build / Buy / Integrate 决策

分析自研、采购、集成开源方案的取舍：

| 方案 | 适合场景 | 优点 | 风险 | 成本 | 上线速度 | 锁定风险 | 推荐度 |
| -- | ---- | -- | -- | -- | ---- | ---- | --- |

不得泛泛推荐产品。必须说明采购或自研的触发条件。

### 11. 不建议清单

列出本次研究主题下最常见但危险的做法，例如：

* 给 Agent 继承用户全部权限
* 让 Agent 直接操作生产系统
* 把外部网页、邮件、文档视为可信上下文
* 只靠系统提示词防提示注入
* 没有工具调用审计
* 没有高风险动作审批
* 没有记忆写入治理
* 没有沙箱和网络隔离
* 没有红队回归测试
* 没有 Agent 身份和责任归因

必须解释每项为什么危险，以及替代方案是什么。

### 12. 结论与决策建议

用直接、明确的方式回答：

* 现在是否应该投入本次研究主题对应的安全建设？
* 最先保护哪类 Agent？
* 最先封堵哪条风险链？
* 最小可行投入是什么？
* 三个月内可以交付什么？
* 哪些风险无法完全消除，只能降低概率和爆炸半径？
* 决策层、架构团队、安全团队、业务团队分别该做什么？

## 八、输出风格

使用简体中文。语言必须精炼、高密度、直接。不要写空泛背景，不要写宣传语，不要堆概念。每一段都必须服务于判断、设计、落地或取舍。

**禁止：**

* 泛泛而谈“加强安全意识”
* 只给原则不给实现
* 只讲威胁不给防线
* 只讲架构不给验收
* 只讲工具不给流程
* 只讲未来趋势不给当下行动
* 编造数据、案例、来源
* 输出可直接用于攻击的代码、绕过步骤、真实目标攻击流程

## 九、自检标准

报告完成前，请自检并修正：

1. 是否始终围绕本次研究主题，而不是跑到泛 AI 安全？
2. 是否体现 AI Agent 特有风险：工具、动作、权限、记忆、规划、自主性、跨系统协作？
3. 是否每个关键结论都有权威来源或明确假设？
4. 是否至少包含 3 个真实落地场景？
5. 是否把风险映射到架构控制点？
6. 是否给出了可执行路线图？
7. 是否能让安全团队、平台团队和业务团队知道下一步做什么？
8. 是否说明了成本、限制、副作用和不可消除风险？
9. 是否删除了没有决策价值的废话？
10. 是否避免了可被滥用的攻击细节？

**请开始深度研究，并输出最终报告。**
