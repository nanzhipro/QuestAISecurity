## 3. 【研究主题】风险层

请建立 AI Agent【研究主题】风险图谱，至少包括：

1. 直接敏感数据外泄：
   - PII、PHI、PCI、财务数据、合同、源代码、密钥、客户数据、内部策略、商业计划、训练数据、模型提示词、系统配置等。

2. 间接敏感信息泄露：
   - 通过摘要、重写、推理、对比、聚合、上下文拼接泄露。
   - 即使原文没有直接输出，模型仍可能泄露推断性敏感信息。

3. 权限过度导致的数据暴露：
   - Agent 使用用户权限、服务账号、API token、OAuth scope、MCP tool 权限读取超出任务需要的数据。

4. Prompt Injection / Indirect Prompt Injection：
   - 外部网页、邮件、文档、图片、代码注释、日历邀请、知识库条目中隐藏恶意指令。
   - Agent 将不可信内容误认为高优先级指令，导致读取或外发敏感数据。

5. Tool / Skill / MCP 风险：
   - Tool metadata 投毒。
   - Tool description 注入。
   - Tool 参数越权。
   - Tool 返回内容携带恶意指令。
   - MCP Server 未认证、未授权、未隔离。
   - Tool chain 中发生 confused deputy 问题。

6. RAG 与知识库风险：
   - 过度召回。
   - 权限过滤缺失。
   - 向量相似度召回跨权限内容。
   - 文档分片导致标签丢失。
   - embedding 中包含敏感信息。
   - RAG poisoning。
   - 引用来源不可追踪。

7. Memory 与日志风险：
   - Agent 长期记忆保存敏感数据。
   - Trace、debug log、prompt log、tool log、eval log 泄露。
   - 供应商观测平台、APM、LLMOps 平台收集敏感上下文。
   - 多租户日志隔离不当。

8. Multi-Agent 风险：
   - Agent 间转发敏感数据。
   - 下游 Agent 权限弱于上游却接收到高敏数据。
   - 委托链失控。
   - Agent swarm 中责任边界不清。

9. 浏览器型/桌面型/代码型 Agent 风险：
   - 浏览器自动填表外泄。
   - 本地文件读取。
   - 代码仓库和密钥扫描不足。
   - CLI Agent 执行命令时泄露环境变量、配置文件和凭据。

10. 供应商与第三方模型风险：
    - Prompt/response 发送到外部模型。
    - 数据被用于训练或改进服务。
    - 跨境传输。
    - 子处理方不透明。
    - API 网关、代理、插件供应链风险。

## 4. 防护控制层

请从“识别、评估、防护、检测、响应、审计、治理”七个阶段设计能力：

### A. 识别

1. 敏感数据发现：
   - 正则、字典、指纹、文档指纹、数据库扫描、代码密钥扫描、NLP/ML 分类、LLM 辅助分类。
2. 数据分类分级：
   - P0/P1/P2/P3 或公开/内部/敏感/高度敏感/受监管。
3. AI Agent 数据资产目录：
   - Agent、Skill、Tool、MCP Server、RAG index、knowledge base、memory store、log store、model endpoint、API key、OAuth app、service account。
4. 数据流发现：
   - Prompt flow、tool flow、retrieval flow、egress flow、agent-to-agent flow。
5. 影子 AI / 影子 Agent 发现：
   - 未登记 Copilot、浏览器插件、SaaS AI、内部脚本 Agent、开发者自建 Agent。

### B. 风险评估

1. 建立 AI Agent 数据风险评分模型：
   - 数据敏感度
   - 暴露范围
   - Agent 自主级别
   - Tool 危险等级
   - 外部连接数量
   - 身份权限强度
   - 是否跨租户/跨境/跨供应商
   - 是否有人类审批
   - 是否有审计和回放
   - 是否可写入外部系统
2. 建立 Agent blast radius 评估：
   - 一个 Agent 被 prompt injection 控制后，最多能读取什么？能写什么？能发到哪里？
3. 建立数据泄露攻击链模型：
   - Source → Agent context → Reasoning/Planning → Tool call → Sink。
4. 建立优先级矩阵：
   - 高价值数据 × 高权限 Agent × 高外发能力 × 低可见性 = 最高优先级。

### C. 防护

1. Prompt 输入侧防护：
   - 敏感输入检测。
   - 隐私提示。
   - 用户侧脱敏。
   - prompt injection 检测。
   - 不可信内容隔离。
2. RAG 防护：
   - 权限感知检索。
   - 文档级、段落级、字段级标签继承。
   - 检索前授权。
   - 检索后重排过滤。
   - 引用和溯源。
3. Tool/Skill 防护：
   - Tool allowlist。
   - 参数级策略。
   - 最小权限。
   - 高风险工具人类确认。
   - 写操作审批。
   - 外发工具限流和阻断。
4. 输出侧防护：
   - response DLP 扫描。
   - 语义泄露检测。
   - 摘要泄露检测。
   - 受监管数据 redaction、masking、tokenization。
   - 根据接收方、通道、租户、目的动态决策。
5. Memory/Log 防护：
   - 默认不持久化敏感上下文。
   - 敏感字段结构化脱敏。
   - trace 分级访问。
   - 保留周期控制。
   - 取证留痕与隐私最小化平衡。
6. Identity & Access 防护：
   - Agent identity。
   - Non-human identity 管理。
   - OAuth scope 最小化。
   - just-in-time access。
   - step-up authentication。
   - service account 风险检测。
7. Runtime Enforcement：
   - AI Gateway。
   - MCP Gateway。
   - sidecar/SDK。
   - reverse proxy。
   - browser isolation。
   - endpoint agent。
   - SaaS API connector。
   - LLM firewall。
8. Egress Control：
   - 邮件、Slack/Teams、Webhook、HTTP request、浏览器提交、文件下载、数据库写入、CRM 更新、工单外发、代码提交等通道的策略执行。
9. Human-in-the-loop：
   - 高敏数据、高风险工具、高影响动作必须支持审批、解释、回放和撤销。

### D. 检测

1. 行为异常检测：
   - Agent 在非典型时间访问大量敏感数据。
   - Agent 调用与任务无关的工具。
   - Agent 读取敏感数据后立即调用外发工具。
   - Agent 输出与用户请求不匹配。
2. 数据流异常：
   - 高敏数据从内部数据源流向外部模型或外部 SaaS。
   - RAG 召回范围异常扩大。
   - 多 Agent 链路中敏感数据跨边界传播。
3. Prompt injection 信号：
   - 忽略先前指令、泄露系统提示、读取秘密、外发数据、隐藏内容指令、编码绕过、多语言绕过等。
4. Tool misuse 信号：
   - 参数异常、频率异常、目的异常、跨权限异常、危险组合调用。
5. 语义泄露信号：
   - 输出没有直接出现原文，但包含可推断的商业秘密、客户隐私、统计小样本隐私或受保护信息。

### E. 响应

1. 阻断、脱敏、降级、审批、沙箱执行、只读模式、撤销 token、禁用 Skill、隔离 Agent。
2. 自动生成事件报告：
   - 涉及数据、主体、Agent、Skill、工具、模型、通道、时间线、策略命中、处置动作。
3. 与 SIEM/SOAR/XDR/ITSM 集成。
4. 支持回放 Agent trace，重建泄露链路。
5. 支持合规通知和数据主体影响评估。

### F. 审计

1. 谁让 Agent 做了什么？
2. Agent 读取了哪些数据？
3. 使用了哪些工具？
4. 哪些数据进入了模型上下文？
5. 哪些数据被输出、脱敏、阻断或审批？
6. 哪个策略生效？
7. 事件能否复现？
8. 是否能生成面向审计、法务、CISO、业务负责人的不同报告？

### G. 治理

1. AI Agent 注册制度。
2. Skill 上架审批。
3. 数据分类分级制度。
4. 高风险 Agent 评审。
5. 安全基线。
6. 开发者安全规范。
7. 红队测试。
8. 持续评估。
9. 供应商风险管理。
10. 合规映射。

# 六、防御性编程与开发者落地

请输出面向开发团队的防御性编程指南。

必须包含：

1. Agent 开发安全原则：
   - 最小权限
   - 不信任外部内容
   - 不将工具返回内容当作指令
   - 数据目的限制
   - 默认不持久化敏感上下文
   - 高风险动作显式确认
   - 所有工具调用可审计
   - 输出前二次检查

2. 安全 API 设计：
   - classify(data)
   - inspect_prompt(prompt, context)
   - inspect_response(response, recipient)
   - inspect_tool_call(tool, args, agent, user)
   - inspect_tool_result(tool, result)
   - inspect_retrieval(query, documents, user, agent)
   - redact(data, policy)
   - authorize(agent, action, resource, purpose)
   - log_event(event)
   - open_incident(event)

3. 安全中间件：
   - LLM middleware
   - Tool middleware
   - RAG middleware
   - MCP middleware
   - Memory middleware
   - Logging middleware

4. 安全编码模式：
   - tool call 前置授权
   - tool result 降权处理
   - untrusted content quarantine
   - structured output validation
   - output schema enforcement
   - sensitive field tagging
   - policy-as-code
   - break-glass access
   - dry-run mode
   - approval token

5. 不安全反模式：
   - 使用全局 admin token 运行 Agent。
   - 把系统提示词、密钥、内部策略暴露给模型。
   - 工具返回内容直接进入下一步指令。
   - RAG 检索不做权限过滤。
   - 日志保存完整 prompt 和 response。
   - Agent 能同时读取敏感数据并访问任意外发工具。
   - 生产环境开启无限制 debug。
   - 多租户共享 memory 或 vector index。
   - 未登记 Skill 可被动态调用。
   - 没有审批的写操作和外发操作。

请给出伪代码、接口示例、策略示例和工程落地建议，但不要提供可用于攻击的 exploit 代码或真实恶意 payload。
